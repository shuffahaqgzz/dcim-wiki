# Goal: Bridge v4.2 Pipeline Architecture Gaps (P1 + P2)

## `/goal` Command

```markdown
/goal Implementasi 7 action items P1+P2 dari gap analysis v4.2-pipeline-architecture untuk mendekatkan arsitektur aktual ke reference design DCIM-Wiki.

First action: Baca dokumen berikut dan laporkan jumlah item per prioritas:
- ~/dcim-wiki/technical-requirements/v4.2-pipeline-architecture-komparasi.md (Section 8.2 & 8.3)
- ~/dcim-wiki/reference-designs/block2-data-ingestion-integration.md (Section 3, 13)
- ~/dcim-wiki/reference-designs/block1-infrastructure-provisioning.md (Section Vault, Monitoring)
Laporkan: jumlah P1 items, jumlah P2 items, komponen yang terdampak. Tunggu konfirmasi sebelum mulai.

Scope:
- IN: Kafka broker HA, TLS, Vault, Schema Registry, Prometheus+Grafana, missing consumers, validation enhancement
- OUT: P3 items (PG/ES upgrade, RBAC, lineage), P4 items, perubahan dokumen existing, deployment ke production

Constraints:
- Tidak mengubah dokumen existing di ~/dcim-wiki/
- Semua konfigurasi baru harus backwards-compatible dengan v4.2 pipeline
- Vault digunakan untuk semua secrets (tidak ada env var baru)
- Kafka TLS harus support TLS 1.2+ minimum
- Schema Registry harus BACKWARD compatibility
- Setiap perubahan harus bisa di-rollback
- Tidak ada downtime untuk pipeline yang sudah berjalan

Done when:
1. File konfigurasi Kafka broker tambahan (2 broker) dibuat di ~/dcim-wiki/technical-requirements/kafka-ha-config/ dengan docker-compose + broker configs
2. File konfigurasi Kafka TLS dibuat di ~/dcim-wiki/technical-requirements/kafka-tls-config/ dengan ssl.properties + keystore/truststore scripts
3. File konfigurasi Vault untuk DCIM secrets dibuat di ~/dcim-wiki/technical-requirements/vault-config/ dengan policy + init script
4. File konfigurasi Schema Registry dibuat di ~/dcim-wiki/technical-requirements/schema-registry-config/ dengan docker-compose + schema examples
5. File konfigurasi Prometheus + Grafana dibuat di ~/dcim-wiki/technical-requirements/monitoring-config/ dengan docker-compose + alert rules + dashboard JSON
6. Dokumen consumer specs untuk SIEM dan Analytics dibuat di ~/dcim-wiki/technical-requirements/missing-consumers-spec.md dengan consumer group config + Kafka topic mapping
7. Dokumen enhanced validation rules dibuat di ~/dcim-wiki/technical-requirements/enhanced-validation-spec.md dengan schema validation + type check + range check rules
8. Summary document dibuat di ~/dcim-wiki/technical-requirements/bridge-implementation-summary.md berisi: implementation checklist, rollback steps, verification commands

Stop if:
- Ada konfigurasi yang memerlukan perubahan ke pipeline production yang sudah berjalan
- Vault setup memerlukan root token atau unseal key yang tidak bisa di-rollback
- Kafka TLS memerlukan CA certificate yang belum tersedia
- Schema Registry memerlukan perubahan ke event schema yang sudah ada
- Konfigurasi monitoring memerlukan agent install di production hosts
- Ada konflik dengan konfigurasi v4.2 yang sudah ada

Use a token budget of 150K tokens for this goal.
```

## Design Choices

| Choice | Rationale |
|--------|-----------|
| **Scope: P1+P2 only** | P1 (Kafka HA, TLS, Vault) adalah kritis untuk data durability. P2 (Schema Registry, Monitoring, Consumers, Validation) adalah high-value improvements. P3/P4 ditunda karena effort lebih tinggi. |
| **Output ke technical-requirements/** | Semua deliverables berupa konfigurasi + dokumentasi, bukan deployment langsung. Tim bisa review sebelum implement. |
| **First action: read + report** | Memastikan pemahaman yang benar sebelum mulai. Pattern ini efektif untuk SDD-style goals. |
| **Rollback-first thinking** | Setiap konfigurasi harus bisa di-rollback. Ini penting untuk production systems. |
| **Token budget 150K** | Multi-component goal dengan 8 deliverables. Budget cukup untuk deep implementation. |
