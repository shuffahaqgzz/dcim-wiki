## 2. Architecture Pattern Comparison

### 2.1 Reference Design Pattern

```
Sources → NiFi (ingest) → Kafka raw → Validation → Kafka validated/enriched
                                              ↓ (fail)
                                            Kafka DLQ → Retry → DLQ

Kafka enriched → Router → CMDB / Asset / Time-Series / SIEM

Cross-cutting: Lineage Tracker (logs every step)
```

**Characteristics:**
- NiFi as primary ingestion tool (100+ processors)
- Centralized validation/enrichment
- Multiple Kafka topics for different stages
- Dedicated stream processing (Flink)
- Formal data lineage tracking

### 2.2 Actual Implementation Pattern

```
Sources → Telegraf (ingest) → Kafka raw → Normalizer → Kafka normalized
                                                           ↓
                                                    NiFi (enrichment only)
                                                           ↓
                                                    Kafka enriched → Consumers
                                                           ↓
                                            PostgreSQL / Elasticsearch / iTop
```

**Characteristics:**
- Telegraf as primary ingestion tool (5 collectors)
- Normalizer for format standardization
- NiFi ONLY for enrichment (not ingestion)
- No dedicated stream processing
- Structured logging for observability

### 2.3 Pattern Comparison Matrix

| Aspect | Reference | Actual | Gap Level |
|--------|-----------|--------|-----------|
| **Ingestion Tool** | NiFi | Telegraf | ⚠️ Different approach |
| **Ingestion Role** | NiFi handles all protocols | Telegraf handles all protocols | ⚠️ Different |
| **Normalization** | In NiFi flow | Dedicated normalizer service | ✅ Similar |
| **Enrichment** | NiFi LookupRecord | NiFi LookupRecord | ✅ Match |
| **Stream Processing** | Flink for real-time | None | ❌ Missing |
| **Data Lineage** | Dedicated tracker | Structured logging | ❌ Missing |

### 2.4 Assessment

**The actual implementation uses a "Telegraf-centric" architecture rather than "NiFi-centric" as designed.**

This is a **deliberate architectural decision** that trades:
- ✅ **Simplicity**: Telegraf is simpler to configure than NiFi for basic polling
- ✅ **Reliability**: Telegraf is battle-tested for metric collection
- ⚠️ **Flexibility**: NiFi's 100+ processors are not available
- ⚠️ **Real-time**: No Flink stream processing
- ⚠️ **Lineage**: No formal lineage tracking

|--------|-----------------|---------------|-----|----------|
| **Schema Definition** | JSON Schema with mandatory fields | Telegraf JSON format → CDM | ⚠️ No formal schema | P2 |

## 5. Technology Stack Comparison

### 5.1 Stack Comparison

| Component | Reference Design | Actual (v4.2) | Recommendation |
|-----------|-----------------|---------------|----------------|
| **Ingestion** | NiFi (100+ processors) | Telegraf (5 collectors) | Keep Telegraf (simpler) |
| **Message Broker** | Kafka (3 brokers, KRaft) | Kafka (1 broker) | Add 2 more brokers for HA |
| **Normalization** | In NiFi flow | `dcim-normalizer.service` | Keep (clean separation) |
| **Enrichment** | NiFi + Enrichment API | NiFi + Enrichment API + Redis | Keep (good implementation) |
| **Stream Processing** | Flink | None | Add for P1/P2 alerts |
| **Storage** | PostgreSQL + ES | PostgreSQL + ES | Keep |
| **Dashboard** | Grafana + Kibana | Kibana only | Add Grafana for metrics |
| **Alerting** | Prometheus + Grafana | Threshold + Telegram | Keep + add Prometheus |
| **Schema Management** | Confluent Schema Registry | metric_mapping.json | Add Schema Registry |
| **Lineage** | Dedicated tracker | Structured logging | Add lineage tracking |
| **Security** | Vault, RBAC, TLS | Env vars, basic auth | Add Vault, RBAC, TLS |

### 5.2 Architecture Pattern Assessment

**Current Pattern: "Telegraf + Kafka + NiFi Hybrid"**

This is a **pragmatic, working architecture** that:
- ✅ Handles current source systems effectively
- ✅ Provides data quality via enrichment
- ✅ Has good alerting (threshold + Telegram)
- ✅ Supports AI training data pipeline (L13/L14)

**Missing from Reference Design:**
- ❌ No real-time processing (< 1s) for P1 alerts
- ❌ No formal schema management
- ❌ No data lineage tracking
- ❌ No HA for Kafka (single broker)

| Component | HA Status | Risk |
|-----------|-----------|------|
| **Kafka** | ❌ Single broker | Data loss on failure |
| **PostgreSQL** | ⚠️ Docker container | Limited HA |
| **Elasticsearch** | ⚠️ Single node | Limited HA |
| **NiFi** | ⚠️ Docker container | Limited HA |

| Priority | Action | Rationale |
